## 先看工具链，不要先猜“智能”

Agent 场景里很容易把注意力都放在提示词和模型输出上，但很多真正决定风险的点其实在更朴素的地方：

- 浏览器工具能访问哪些域；
- 文件读写边界在哪里；
- MCP 服务器暴露了哪些动作；
- 工具调用结果会不会原样回流进后续推理。

先把这些工具边界摸清楚，比直接讨论“模型会不会被骗”更有落点。

## 我会先记录一条最短调用链

例如某个 Agent 会这样工作：

```text
用户输入 -> Planner 生成步骤 -> Browser 打开页面 -> Tool 返回文本 -> Agent 再决定下一步
```

真正该分析的是：

1. 哪一步首次接收了不可信数据；
2. 哪一步把这些数据又当成“可信上下文”继续喂给后续决策；
3. 哪一步拥有足够强的副作用，例如下载、点击或写文件。

## 浏览器自动化里的风险点

在浏览器工具链里，我最在意这几件事：

### DOM 文本和指令的界线

页面文本只是内容，还是会被后续步骤误当成操作建议？如果工具结果被直接拼进下一轮提示里，就要重点看这里。

### 页面跳转后的作用域变化

打开同站点页面和被跳到第三方页面，风险完全不是一个量级。域切换时有没有重新确认允许范围，是很关键的分界线。

### 下载与本地落地

只要工具链具备“看见内容 -> 下载文件 -> 在本地继续处理”的能力，就必须把来源、命名、落地路径和后续解析动作一起记清楚。

## 复现时我更想留下什么

我更想留下的是这种可重放的观察序列：

```text
1. 让 Agent 打开受控页面
2. 页面返回一段带有误导性文本的内容
3. 记录 browser / tool 的原始返回
4. 比较下一步动作有没有受到影响
```

只要这条序列可重复，后面无论是做防守分析还是写实验记录，都会更稳。

## 一个更适合博客的写法

与其把 Agent 渗透写得很玄，不如把它写成：

> 这次实验里，真正的关键不是“模型懂了什么”，而是哪一个工具结果在没有重新校验的情况下，被当成了后续动作的决策依据。

这种写法更接近工程系统，也更方便继续扩展到下一篇实验里。
