## 先确认“谁在刷新”

会话漂移问题很烦人的一点，是你经常能看到 401，却不确定到底是谁在负责刷新：

- 前端请求拦截器；
- 某个 SDK；
- 网关层；
- 服务端隐式续期逻辑。

如果不先把责任点找出来，抓到再多请求也只是噪音。

## 我会先做一条最短时间线

我通常先只保留下面四类事件：

1. 业务请求发出；
2. 返回 401；
3. 刷新请求出现；
4. Cookie 或 Authorization 头变化。

一旦时间线被压干净，很多“偶发问题”都会开始变得有规律。

## 重点看两个错位

### 刷新成功，但旧请求没有被正确重放

这类问题表面像“刷新没生效”，实际上常常是旧请求已经失败结束，新的 token 根本没机会带回去。

### token 更新了，但浏览器状态没同步

如果内存态、Cookie 和本地存储同时存在，就要确认到底哪个是下一跳真正会读取的值。很多 401 抖动不是刷新失败，而是读取来源不一致。

```ts
if (response.status === 401 && !request.meta?.retried) {
  await refreshToken();
  request.meta = { ...request.meta, retried: true };
  return http(request);
}
```

这段代码看起来标准，但如果 `refreshToken()` 只更新了某一个状态源，或者重试请求复用了旧头，就还是会抖动。

## 记录时我会保留哪些证据

- 首次 401 对应的是哪一个资源；
- 刷新请求发出前后的 Cookie 差异；
- 重放请求是否真的换了 header；
- 第二次失败是不是同一原因。

这些证据比一句“token 有问题”更能帮你下次快速定位。

## 最后落成一句话

我更愿意把结论写成：

> 当前问题的根因不是刷新接口本身失败，而是刷新后的状态同步和原请求重放没有形成同一条稳定链路。

这样不管后面是改前端还是改网关，讨论都会更清楚。
