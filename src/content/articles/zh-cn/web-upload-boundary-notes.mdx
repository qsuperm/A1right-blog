## 先把“真实执行流”跑通

做文件上传题或者看真实业务时，最容易犯的错，就是一上来先堆 payload，结果连服务端到底按什么顺序校验都没看清。

我现在更习惯先确认四件事：

1. 浏览器端有没有只是“提示型”校验；
2. 反向代理有没有先拦一层；
3. 应用层的 MIME、扩展名、文件头校验顺序是什么；
4. 最终落盘路径和访问路径是不是同一条线。

只有这条链路被确认下来，后面的绕过才不会变成盲猜。

## 最常见的四个拐点

### 1. 前端限制不等于后端限制

很多页面会把 `accept="image/*"` 做得很显眼，但这通常只是降低普通用户误操作，不是安全边界。真正的边界还是服务端路由里的解析和持久化逻辑。

### 2. MIME 判断和扩展名判断经常脱节

有些服务先看扩展名，再由对象存储或中间件补一次 MIME；也有些链路是反过来的。只要两处判断依据不同，就要重点看“哪一层的结果最终决定是否放行”。

```ts
const allowExt = ['png', 'jpg', 'jpeg'];
const ext = filename.split('.').pop()?.toLowerCase();

if (!ext || !allowExt.includes(ext)) {
  throw new Error('blocked by ext');
}

if (!contentType.startsWith('image/')) {
  throw new Error('blocked by mime');
}
```

这段逻辑本身不复杂，但如果重命名发生在后面，或者对象存储又做了一次内容探测，执行结果就可能和第一眼看到的不一样。

### 3. 重命名比后缀更关键

只看“能不能上传某后缀”还不够，还要看落盘前会不会被统一改名。如果最终文件名被改成随机 UUID，再配合固定图片目录，很多直接访问型风险会立刻收缩。

### 4. 可访问路径决定风险等级

能上传，不代表能执行；能落盘，不代表能访问；能访问，也不一定能以预期的解释器方式执行。风险判断一定要落在最终访问效果上。

## 我会怎么做记录

我会把每一层的判断压成一张很短的表：

| 层级 | 观察点 | 结果 |
| --- | --- | --- |
| 前端 | `accept` / JS 提示 | 仅体验限制 |
| 代理 | 请求体限制 | 需要确认 |
| 应用 | 扩展名 + MIME | 真正拦截点 |
| 存储 | 重命名 / 目录 | 影响回显与访问 |

这样后面复盘时，不会忘记“到底是哪一层决定了结果”。

## 一个更稳的结论方式

与其直接说“这里能不能绕”，不如把结论写成：

> 当前链路里，真正影响上传结果的是应用层扩展名校验和落盘前重命名；浏览器提示和表面 MIME 不足以单独构成安全边界。

这样的结论更像工程记录，也更方便下一次继续接着看。
